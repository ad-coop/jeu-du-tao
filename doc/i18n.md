# i18n — Internationalisation

## Architecture

Translations are served by the backend as a flat JSON map:

```
GET /api/i18n/{locale}  →  { "key": "value", … }
```

The frontend fetches this map on startup (`I18nProvider`) and exposes a `t(key, params?)` function via React context.
French (`fr`) is the only supported locale for now.

**Source files:**
- `app-backend/src/main/resources/messages_fr.properties` — all French translations
- `app-frontend/src/i18n/I18nProvider.tsx` — fetch + context
- `app-frontend/src/i18n/useTranslation.ts` — hook

## Key Naming Convention

```
<feature>.<section>.<element>[.<qualifier>]
```

| Segment | Examples |
|---------|---------|
| feature | `landing`, `game` |
| section | `create`, `join`, `waiting`, `error` |
| element | `title`, `submit`, `handle`, `players` |
| qualifier | `label`, `placeholder`, `hint`, `copy`, `required` |

**Current prefixes (spec 0002):**

| Prefix | Used for |
|--------|---------|
| `landing.*` | Landing page |
| `game.create.*` | Game creation form |
| `game.join.*` | Join game form |
| `game.waiting.*` | Waiting room |
| `game.error.*` | Error page messages |
| `game.create.error.*` | Validation errors on creation form |
| `game.join.error.*` | Validation errors on join form |

## Interpolation

The `t()` function supports `{param}` placeholders:

```typescript
t("game.waiting.kick.label") + " " + name
// or with interpolation:
t("game.waitingRoom.kick", { name: "Alice" })  // → "Expulser Alice"
```

## Workflow — Adding New Keys

**The frontend is the single source of truth for key names.**

1. Write the `t("some.key")` call in the frontend component.
2. Add the corresponding entry to `messages_fr.properties` using the **exact same key**.
3. Verify with: `grep -rh 't("' app-frontend/src --include="*.tsx" --include="*.ts" | grep -oP '(?<=t\(")[^"]+' | sort -u`

> ⚠️ Never invent key names in the properties file first — they will diverge from what the frontend calls.

## Parallelism Rule

When backend and frontend are implemented by separate agents in parallel:
- The **frontend agent** owns i18n end-to-end: it writes both the `t()` calls **and** the `messages_fr.properties` entries.
- The backend agent does not touch translation keys.
